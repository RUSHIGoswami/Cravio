# ADR-0001: Single monorepo with per-package CLAUDE.md

- **Status:** Accepted
- **Date:** 2026-06-20
- **Deciders:** Founder
- **Phase:** Phase 0

## Context
Cravio is built solo with AI agents across a backend, a mobile app, an admin/web dashboard, and infrastructure. Agents need local, reliable context per area, and the API contract must stay in lockstep with the client that consumes it.

## Decision
Use a single monorepo with top-level packages `/api`, `/mobile`, `/admin`, `/infra`, `/docs`. Each package owns a `CLAUDE.md` with its conventions, entry points, and run/test commands. The OpenAPI contract is generated by `/api` and committed under `/docs` for `/mobile` and `/admin` to consume.

## Rationale
- One PR can change the API and its mobile consumer together, keeping the contract honest.
- Per-package `CLAUDE.md` gives each agent session tight, relevant context without loading the whole tree.
- A single CI pipeline and one dependency-review surface is cheaper to operate for a solo founder.

## Alternatives considered
- **Polyrepo (separate repos per package):** cleaner isolation, but contract drift between API and mobile becomes a constant tax, and cross-cutting agent tasks span repos. Rejected for a solo build.
- **Nx/Turborepo tooling:** powerful but adds build-graph complexity we don't need pre-revenue. Revisit if CI times grow.

## Consequences
- CI must scope jobs by changed package to keep runs fast.
- Requires discipline: shared code goes through explicit interfaces, not deep cross-imports.
- The committed OpenAPI spec becomes a build artifact that must be regenerated on every API change (enforced in CI).
